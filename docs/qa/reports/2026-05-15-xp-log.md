## 2. Defects

### Defect 1 — Multi-buy advantage/flaw undo overshoots intermediate state

- **Severity:** High (correctness of advertised feature on a release branch)
- **Title:** Undoing an advantage purchase made in a multi-buy Spend XP session restores `advantages_remaining` to the *session-start* value instead of the *immediately-prior* value
- **Game lines affected:** VtM (empirically confirmed), Werewolf (source review — same pattern), Hunter (source review — same pattern, slightly different state shape so needs empirical verification)
- **Known to PR author:** yes — flagged in the testing brief as "PR #281: within-session priorValue may be wrong"

#### Steps to reproduce (VtM)

1. Log in. Open an editable VtM character. Make sure baseline Advantage Dots Remaining is known (e.g. `7`) and Remaining XP is enough (e.g. `50`).
2. Open **Spend XP**. Pick category **Advantage**. Select dots = `2`. Click **Purchase**. Don't close yet.
3. In the same dialog, select **Advantage** again, dots = `3`, click **Purchase**.
4. Click **Save** on the Spend XP dialog.
5. Open **XP Log**. Two entries appear, newest first: `Added 3 advantage point(s)` (top) and `Added 2 advantage point(s)` (bottom).
6. Click **Undo** on the top entry, confirm.

#### Expected

- Remaining XP: `35 + 9 = 44` ✓
- Advantage Dots Remaining: `12 − 3 = 9`
- The bottom entry remains, still showing `Added 2 advantage point(s)`.

#### Actual

- Remaining XP: `44` ✓
- **Advantage Dots Remaining: `7`** ✗ (jumped past `9` straight to the session-start baseline)
- Bottom entry remains visible. If the user then undoes it: XP `+6 → 50` (correct), Advantage Dots `→ 7` (no change — already there). The *net* end-state of fully undoing the session is correct (only because both entries restore to the same baseline), but a user partially undoing will see a wrong dot count and wrong XP-vs-dots accounting.

#### Root cause (source review)

- [src/components/character_creator/vtm/spendXp.vue:467-471](src/components/character_creator/vtm/spendXp.vue#L467-L471) — when "Purchase" is clicked for an Advantage, `handlers.advantage.record(this, ...)` is called with `this` as state. The handler reads `this.advantages_remaining`, which was seeded once at dialog open from `props.info.advantages_remaining` (line 279) and **is not mutated during the dialog**. The local accumulator `this.advantagePoints` is the delta and is incremented separately on line 471. So every `record(...)` call within the same dialog sees the same starting `advantages_remaining`, and every recorded entry carries the same `priorValue`.
- [src/lib/xp/handlers/vtm.js:228](src/lib/xp/handlers/vtm.js#L228) — `undo` is `state[entry.payload.counter] = entry.payload.priorValue;`, i.e. it *assigns* the snapshot back rather than reversing the recorded `delta`. When two entries share the same `priorValue`, undoing the top entry overshoots.
- Same pattern in [src/components/character_creator/werewolf/spendXp.vue:539-543](src/components/character_creator/werewolf/spendXp.vue#L539-L543) and the matching `undo` in [src/lib/xp/handlers/werewolf.js:230-232](src/lib/xp/handlers/werewolf.js#L230-L232).
- Same pattern in [src/components/character_creator/hunter/spendXp.vue:346-350](src/components/character_creator/hunter/spendXp.vue#L346-L350) and [src/lib/xp/handlers/hunter.js:153](src/lib/xp/handlers/hunter.js#L153). (Hunter builds a fresh `handlerState` object each call from `this.advantagePoints`, an accumulator that starts at `0` — so behavior in Hunter is slightly different and worth empirically rechecking.)

#### Why the unit tests don't catch it

[src/lib/xp/handlers/\_\_tests\_\_/vtm.test.js:285-301](src/lib/xp/handlers/__tests__/vtm.test.js#L285-L301) and the `flaw` test directly below it only test the *single*-record → single-undo cycle. They manually set `charState.advantages_remaining = 6` between `record` and `undo` to simulate the parent applying the delta — but that simulation never produces two records with the same `priorValue` against a state that has been incremented twice. Suggested test addition:

```js
const charState = { advantages_remaining: 5 };
const e1 = handlers.advantage.record(charState, { delta: 1 });
charState.advantages_remaining = 6;
const e2 = handlers.advantage.record(charState, { delta: 1 }); // priorValue should be 6, will probably be 5
expect(e2.payload.priorValue).toBe(6);
```

…and a parallel integration test where the *dialog component* records two entries without the parent mutating state in between.

#### Suggested fix

Two options:

1. Update `advantage` / `flaw` `undo` to reverse by `delta`: `state[counter] -= entry.payload.delta;` instead of `= priorValue`. This is robust regardless of how `record` captured `priorValue`.
2. Or, in the Spend XP dialog, mutate `this.advantages_remaining` (in addition to `this.advantagePoints`) after each record so the next `record(...)` sees the updated state.

Option 1 is the smaller blast radius. Option 2 also makes the dialog's running display match the post-save reality, which is friendlier.

  That looks like a Heroku-generated DB name baked into the query at some point. In local dev the DB is `login`; the query references a database that does not exist. MySQL surfaces `Unknown database 'ey140u9j4rs9xcib'`, which Sequelize bubbles up as a 500.

- **Console error:** `Failed to load resource: the server responded with a status of 500 () @ http://localhost:5000/favorites/my:0`
- **Also affects:** SQL injection surface here is also notable — `req.currentUser.id` is interpolated raw via template literal rather than a parameter binding. Worth a defensive fix in the same patch.
